# 通知中心设计方案

**目标**：为 YomuTomo 增加一个全站统一的通知中心，通知落库并按用户跨设备同步，导航栏提供未读角标和历史通知面板，打开后自动清空未读数。

## 背景

当前新闻中心和其他页面只依赖 toast 做瞬时提示，消息不会持久化，用户刷新、切换设备或离开页面后就看不到历史记录。对于新闻生成失败这类需要追溯原因的状态，单次 toast 不足以支撑用户回看和跨设备同步，因此需要一个独立的通知中心。

## 设计目标

- 通知必须落库，按用户同步。
- 导航栏 `YomuTomo` 右侧紧贴一个通知铃铛按钮。
- 铃铛右上角显示未读数量角标。
- 点击铃铛打开历史通知面板。
- 打开面板时，当前用户的未读通知自动标记为已读，角标立刻归零。
- 通知中心作为全站统一能力，后续首页生成文章、设置保存成功、系统报错等都可以复用。

## 范围

本次先把统一通知中心能力做出来，并先接入现有新闻任务链路。其他来源后续按同一接口接入，不在本次范围内一次性完成。

## 通知数据模型

新增 `notifications` 表，建议字段如下：

- `id`：主键
- `user_id`：所属用户
- `type`：通知类型，例如 `news_success`、`news_failed`、`article_generated`、`settings_saved`、`system_error`
- `title`：通知标题，给列表第一行使用
- `message`：通知摘要或正文，给列表第二行使用
- `source_task_id`：来源任务 ID，用于追溯
- `source_url`：可选，来源页面或详情页地址
- `is_read`：是否已读
- `read_at`：标记已读时间
- `created_at`：创建时间
- `updated_at`：更新时间

### 幂等要求

同一个来源任务和同一种通知类型不能重复写入。建议加唯一约束或幂等键，优先使用 `user_id + type + source_task_id`，避免任务轮询或重试时重复写通知。

## 通知来源

统一通知中心支持多个来源，但首期只接入现有新闻任务。

- 新闻生成成功：写入成功通知
- 新闻生成失败：写入失败通知
- 未来扩展：
  - 首页文章生成成功
  - 设置保存成功
  - 系统错误

## API 设计

### `GET /notifications`

返回当前用户通知列表和未读数，用于初始化通知面板和角标。

返回结构建议：

```json
{
  "items": [
    {
      "id": 1,
      "type": "news_failed",
      "title": "新闻生成失败",
      "message": "正文抓取失败：未能提取到新闻正文，请稍后重试。",
      "source_task_id": 12,
      "source_url": "/news_center",
      "is_read": false,
      "created_at": "2026-06-08T14:00:00+09:00"
    }
  ],
  "unread_count": 3
}
```

### `GET /notifications/unread-count`

仅返回未读数，供导航栏初始化角标使用。页面加载时可以先用这个接口快速渲染角标，再按需拉取列表。

### `POST /notifications/mark-read`

把当前用户的通知标记为已读。

建议支持两种模式：

- 不传 `notification_id`：标记当前用户全部未读通知为已读
- 传 `notification_id`：只标记单条通知已读

首期需要“打开面板自动清空角标”，所以至少要支持“全部标记已读”。

## 前端交互

### 导航栏按钮

- 在 `YomuTomo` 右侧紧贴位置加一个铃铛按钮
- 按钮使用现有亚克力风格，和当前导航语言一致
- 右上角显示未读数字角标
- 角标在未读数为 0 时隐藏

### 通知面板

- 点击铃铛弹出下拉面板
- 面板显示历史通知，按时间倒序排列
- 每条通知为完整卡片，包含：标题、摘要、时间、来源入口
- 成功和失败通知可以用不同强调色，但整体风格保持统一
- 面板打开后自动调用已读接口，清空角标
- 再次有新通知时，角标重新出现

## 交互细节

- 如果面板打开时网络失败，角标不要提前清零，避免误判
- 如果通知列表为空，展示空状态文案
- 点击通知卡片可跳转到来源页面或详情页，若有 `source_url` 则优先使用
- 点击面板外部区域关闭面板
- 支持 `Esc` 关闭面板

## 后端集成点

- 新闻任务完成后写通知
- 新闻任务失败后写通知
- `/crawl_status` 继续保留任务状态，前端不再把它作为唯一消息来源
- 通知写入逻辑建议放到独立服务或工具函数里，避免散落在多个路由与爬虫函数中

## 样式与布局

- 通知按钮应与当前导航按钮视觉统一，但要保持相对独立的图标外观
- 角标使用小红点/数字胶囊样式，数字超过两位时可显示 `99+`
- 面板宽度适中，桌面端可固定在导航栏下方右侧，移动端可变成全宽抽屉式或卡片式下拉

## 错误处理

- 读取通知失败时，角标和面板都要有明确的兜底文案
- 标记已读失败时，不要静默吞错，要保留错误日志并让用户知道刷新后重试
- 写通知失败时，不能影响主流程的文章生成或设置保存，但要记录日志，方便追踪

## 测试策略

- 模型测试：通知表字段、唯一约束、已读状态切换
- API 测试：
  - 获取未读数
  - 获取通知列表
  - 标记已读后未读数归零
  - 重复写入同一来源通知不会产生重复记录
- 前端测试：
  - 点击铃铛打开面板
  - 打开后角标消失
  - 通知列表正确渲染
- 集成测试：新闻任务成功或失败后会写入通知，并能在通知面板中看到历史记录

## 实施顺序

1. 新增通知表和迁移
2. 新增通知服务与 API
3. 接入新闻任务写通知
4. 在导航栏增加通知按钮和角标
5. 新增通知面板和前端数据加载
6. 补测试并做 Docker 验证
